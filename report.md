# Отчёт по лабораторной работе №1: Нормализация данных в модель «Звезда»

## Введение
В лабораторной работе я преобразовал исходные “плоские” данные о продажах товаров для домашних питомцев из **10 CSV-файлов** (в сумме **10 000 строк**) в аналитическую модель данных типа **«звезда»** в СУБД **PostgreSQL**.  
Цель — получить структуру “факт + измерения”, удобную для аналитических запросов (агрегации по товарам, покупателям, магазинам, датам и т. п.).

---

## Исходные данные
Исходный CSV содержит смешанные сущности в одной строке: данные о покупателе, продавце, магазине, товаре, поставщике и параметрах продажи. По смыслу одна строка соответствует **одной продаже** (sale) с метриками `sale_quantity` и `sale_total_price`.

---

## Проделанная работа

### 1) Загрузка исходных данных в staging-таблицу
Для работы с данными средствами SQL я использовал staging-таблицу:
- `public.mock_data`

Загрузка сделана **автоматически через Docker**:
- `docker-compose.yml` поднимает PostgreSQL
- init-скрипты в `docker/init/` создают таблицу и загружают все CSV в `mock_data` при первом старте контейнера:
  - `docker/init/00_mock_data_ddl.sql`
  - `docker/init/01_mock_data_load.sql`

Такой подход упрощает проверку: достаточно запустить контейнер, и база сразу содержит исходные данные.

### 2) Проектирование модели «звезда»
Я реализовал схему, где в центре находится таблица фактов `fact_sales`, а вокруг — измерения `dim_*`.

**Таблица фактов:**
- `public.fact_sales`
  - хранит ключи на измерения
  - хранит метрики продаж: количество и итоговую стоимость

**Измерения:**
- `public.dim_date` — календарь по `sale_date`
- `public.dim_customer` — покупатели (атрибуты `customer_*`)
- `public.dim_seller` — продавцы (атрибуты `seller_*`)
- `public.dim_store` — магазины (атрибуты `store_*`)
- `public.dim_supplier` — поставщики (атрибуты `supplier_*`)
- `public.dim_product` — товары (атрибуты `product_*` + связь на поставщика)

Таким образом, “плоская” строка из `mock_data` декомпозируется на:
- **измерения** (описание сущностей)
- **факт** (событие продажи + числовые показатели)

DDL для витрины находится в:
- `sql/01_ddl.sql` (создание `dim_*` и `fact_sales`)

### 3) Трансформация и заполнение (DML)
Заполнение выполнено в логике “сначала измерения, потом факт”:
1. `dim_date` — парсинг строковой даты `sale_date` в тип `date`
2. `dim_customer`, `dim_seller` — загрузка по идентификаторам из полей `sale_customer_id`, `sale_seller_id`
3. `dim_store`, `dim_supplier` — загрузка уникальных значений и присвоение surrogate key
4. `dim_product` — загрузка по `sale_product_id`, привязка к `dim_supplier`
5. `fact_sales` — загрузка событий продаж с подстановкой `date_id` и `store_id`

Ключевой практический момент: при связывании магазина использовалось сравнение с учётом `NULL` в отдельных атрибутах (например `store_state`), чтобы не терять строки при join’ах.

DML находится в:
- `sql/02_dml.sql`

## Проверка корректности результата
Для верификации я добавил отдельный скрипт:
- `sql/03_check_success.sql`

Он проверяет:
- количество строк в `mock_data`, измерениях и факте
- разницу между `mock_data` и `fact_sales` (по идее должна быть 0)
- “потери” при подстановке даты и магазина (несопоставившиеся `date/store`)
- ссылочную целостность: нет ли строк факта без соответствующих измерений
- контрольные агрегаты (топ стран/товаров по выручке)

Критерий успеха: `fact_sales` содержит все продажи из `mock_data`, а проверки `not_matched_*` и `bad_*_fk` дают нули.

## Инструкция по запуску
1. Перейти в папку `lab1/BDSnowflake`.
2. Запустить PostgreSQL:

```bash
docker compose up
```

Если контейнер уже запускался ранее и нужно заново выполнить init-скрипты загрузки CSV:

```bash
docker compose down -v
docker compose up
```

3. Подключиться к БД (например, через DBeaver) и выполнить по очереди:
- `sql/01_ddl.sql`
- `sql/02_dml.sql`
- `sql/03_check_success.sql`
